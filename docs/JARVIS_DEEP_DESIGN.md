## 一、JARVIS 状态机设计

JARVIS 的运行由明确的**状态机**驱动，确保各模块协调工作：

```
                          ┌─────────────┐
                          │   SLEEP     │ ← 持续监听麦克风，仅唤醒词引擎工作
                          │  (休眠)      │    功耗最低，无 LLM 调用
                          └──────┬──────┘
                                 │ 检测到 "Javis"
                                 ▼
                 ┌─────────────────────────────┐
          ┌─────▶│    LISTEN                   │
          │      │   (聆听)  录音中...           │
          │      │   Whisper 将语音→文字         │
          │      │   超时 N 秒无声音 → SLEEP     │
          │      └────────────┬────────────────┘
          │                   │ 语音识别完成
          │                   ▼
          │      ┌─────────────────────────────┐
          │      │    THINK                     │
          │      │   (思考)  LLM 推理中          │
          │      │   • 意图识别                  │
          │      │   • 任务规划                  │
          │      │   • 决定下一步行动             │
          │      └────────────┬────────────────┘
          │                   │ 需要工具 / 需要回复
          │          ┌────────┴────────┐
          │          ▼                  ▼
          │  ┌──────────────┐  ┌──────────────┐
          │  │    ACT       │  │   SPEAK      │
          │  │  (执行)       │  │  (说话)       │
          │  │  调用 Tool    │  │  TTS 播放     │
          │  └──────┬───────┘  └──────┬───────┘
          │         │ 工具结果         │
          │         ▼                  │
          │  ┌──────────────┐          │
          │  │  OBSERVE     │          │
          │  │ (观察)        │          │
          │  │ 验证结果      │          │
          │  └──────┬───────┘          │
          │         │ 未完成            │
          │         └──→ 回到 THINK ────┘ (ReAct 循环)
          │
          │                   ┌─────────────────┐
          │                   │ SPEAKING        │
          │                   │ (正在说话) TTS 播放 │
          │                   └────────┬────────┘
          │                            │
          │    用户此时说话 (打断)       │ TTS 播放完毕
          │          │                  │
          │          ▼                  │
          │     ┌──────────┐            │
          │     │  INTERRUPT│            │
          │     │  (打断)    │            │
          │     │ 停 TTS    │            │
          │     └────┬─────┘            │
          │          │                  │
          └──────────┘                  │
                                        ▼
                              ┌──────────────────┐
                              │    IDLE          │
                              │  (空闲) 等待下轮  │
                              └────────┬─────────┘
                                       │
                                       ▼
                                  ┌──────────┐
                                  │  SLEEP   │
                                  └──────────┘
```

### 状态详细说明

| 状态 | 功耗 | LLM 调用 | 麦克风 | 说明 |
|------|------|---------|--------|------|
| **SLEEP** | 💤 极低 | ❌ | ✅ (仅唤醒词) | 只运行 Porcupine 离线检测，其余全停 |
| **LISTEN** | 📡 中 | ❌ | ✅ (全速率) | 环形缓冲+Whisper 识别，最长 30 秒 |
| **THINK** | 🧠 高 | ✅ | ❌ | LLM 推理 + ReAct 决策，可多次往返 |
| **ACT** | ⚡ 高 | ❌ | ❌ | 执行工具 (截图/鼠标/键盘/摄像头) |
| **OBSERVE** | 👀 中 | ✅ | ❌ | 观察工具结果，决定继续还是回复 |
| **SPEAK** | 🔊 低 | ❌ | ✅ (打断检测) | TTS 播放，同时检测用户是否打断 |

### 状态转换规则

```
规则 1: THINK → SPEAK
  当 LLM 决定直接回复用户时

规则 2: THINK → ACT
  当 LLM 决定调用工具时

规则 3: ACT → OBSERVE
  工具执行完毕，分析结果

规则 4: OBSERVE → THINK
  任务未完成，继续 ReAct 循环

规则 5: OBSERVE → SPEAK
  任务完成，回复用户

规则 6: SPEAK → SLEEP
  TTS 播放完毕，回到休眠等待唤醒

规则 7: SPEAK → LISTEN
  用户说话打断，停止 TTS，开始监听
```

## 三、并发模型

JARVIS 需要同时处理多路输入输出，采用 **asyncio 事件驱动架构**：

```
                    ┌─────────────────────────────────────────┐
                    │           asyncio Event Loop             │
                    │                                         │
                    │  ┌──────────────┐                       │
                    │  │ Task 1:      │ ← 主 ReAct 循环        │
                    │  │ Agent Loop   │    (优先级最高)         │
                    │  └──────────────┘                       │
                    │                                         │
                    │  ┌──────────────┐                       │
                    │  │ Task 2:      │ ← 持续监听麦克风       │
                    │  │ Mic Listener │    (后台, 低优先级)    │
                    │  └──────────────┘                       │
                    │                                         │
                    │  ┌──────────────┐                       │
                    │  │ Task 3:      │ ← 打断检测             │
                    │  │ Interrupt    │    (TTS播放时监听)      │
                    │  │ Detector     │                       │
                    │  └──────────────┘                       │
                    │                                         │
                    │  ┌──────────────┐                       │
                    │  │ Task 4:      │ ← TTS 播放队列         │
                    │  │ TTS Player   │    (独立播放,不阻塞)   │
                    │  └──────────────┘                       │
                    │                                         │
                    │  ┌──────────────┐                       │
                    │  │ Task 5:      │ ← 工具执行（异步）     │
                    │  │ Tool Executor│    (截图/摄像头等)     │
                    │  └──────────────┘                       │
                    └─────────────────────────────────────────┘
```

### 并发优先级

```
高优先级 (必须立即响应):
  ▸ 唤醒词检测任务     → 实时
  ▸ 打断检测任务       → TTS 播放期间必须监听
  ▸ 主 ReAct 循环      → 用户交互的主线程

中优先级 (影响体验):
  ▸ TTS 播放           → 语音反馈
  ▸ 工具执行           → 截图/摄像头/鼠标

低优先级 (后台):
  ▸ 系统状态监控
  ▸ 日志写入
  ▸ 记忆系统写入
```

### 线程/进程模型

```
┌──────────────────────────────────────────────┐
│                  主进程 (Main Process)          │
│                                              │
│  ┌──────────────┐   asyncio Event Loop       │
│  │  Agent Loop   │ ← 整个 ReAct 在同个协程     │
│  │  (协程)       │    避免锁竞争               │
│  └──────────────┘                            │
│                                              │
│  非阻塞操作 → 直接 await                     │
│    • screenshot()    → await                │
│    • camera_snap()   → await                │
│    • LLM API 调用     → await                │
│                                              │
│  阻塞操作 → 跑在 executor (线程池)           │
│    • whisper ASR     → run_in_executor       │
│    • PaddleOCR       → run_in_executor       │
│    • pyautogui 鼠标   → run_in_executor       │
│    • OpenCV 帧捕获   → 独立线程               │
│                                              │
│  ┌──────────────────┐                        │
│  │ 线程池 (ThreadPool) │ ← CPU 密集型/阻塞IO   │
│  │ max_workers=4     │                       │
│  └──────────────────┘                        │
└──────────────────────────────────────────────┘
```

## 四、错误处理链

### 分层错误恢复策略

```
Level 1: 工具级 (自动重试)
  ┌────────────────────────────────────────┐
  │ 工具执行失败 → 自动重试 3 次            │
  │                                        │
  │ 适用场景:                               │
  │ • 截图失败 → 重试截图                   │
  │ • 鼠标点击无反应 → 重试点击              │
  │ • 摄像头无响应 → 切换摄像头索引           │
  │ • 网络超时 → 指数退避重试                │
  │                                        │
  │ 模式: 静默重试, 不打扰用户               │
  └────────────────────────────────────────┘
        重试 3 次仍失败
              ▼
Level 2: 工具级 (降级策略)
  ┌────────────────────────────────────────┐
  │ 工具失效 → 启用降级替代方案              │
  │                                        │
  │ 场景 | 降级方案                         │
  │ 摄像头不可用 → 告知用户，跳过摄像头      │
  │ 鼠标控制失败 → 换 pyautogui 替代        │
  │ 截图全屏失败 → 截取当前活动窗口          │
  │ OCR 失败 → 发送截图让 LLM Vision 读     │
  │ LLM API 超时 → 切 fallback 模型         │
  └────────────────────────────────────────┘
        降级也失败
              ▼
Level 3: 智能体级 (LLM 决策)
  ┌────────────────────────────────────────┐
  │ 报告给 LLM，让它决定下一步              │
  │                                        │
  │ 格式: ToolError(name, reason, suggest)  │
  │ LLM 可:                                 │
  │ • 换一种方式完成任务                     │
  │ • 请求用户帮助                          │
  │ • 放弃并解释                            │
  └────────────────────────────────────────┘
        用户交互
              ▼
Level 4: 用户级 (最终兜底)
  ┌────────────────────────────────────────┐
  │ Javis: "抱歉，我无法打开摄像头，         │
  │         可能被其他应用占用了。            │
  │         你要不检查一下摄像头权限？"       │
  └────────────────────────────────────────┘
```

### 具体错误场景处理

| 错误 | 处理方式 |
|------|---------|
| **Wake word 误触发** | 静默超时 (3 秒无语音 → 回到 SLEEP) |
| **ASR 识别失败** | "没听清，请再说一遍" |
| **LLM API 超时** | 降级到更小模型 / 重试带退避 |
| **工具执行异常** | 重试 → 降级 → 报告 LLM |
| **摄像头被占用** | 提示用户关闭占用应用 |
| **连续失败 N 次** | 暂停操作，请求用户指导 |
| **配置修改冲突** | 保留旧值，提示格式错误 |

## 五、安全与权限模型

### 权限等级划分

```
Level 0: 只读 — 无副作用
  ├── 摄像头拍照 (仅捕获不存储)
  ├── 屏幕截图
  ├── 文件读取
  ├── 剪贴板读取
  ├── 系统信息查询
  ├── 网络搜索
  └── 配置读取

Level 1: 低风险 — 可自动执行
  ├── 语音回复 (TTS)
  ├── 剪贴板写入
  ├── 文件写入 (仅指定目录)
  └── 打开已知安全应用 (Chrome/记事本)

Level 2: 中风险 — 需确认
  ├── 鼠标点击 (非用户主动指示时)
  ├── 键盘输入 (自动填入内容)
  ├── 文件删除
  ├── 打开非白名单应用
  └── 执行系统命令

Level 3: 高风险 — 严格禁止
  ├── 读取密码/密钥文件
  ├── 执行 rm -rf / 类命令
  ├── 修改系统关键设置
  ├── 下载并执行未知程序
  └── 模拟用户登录
```

### 工具执行安全检查

```
每个工具调用在 dispatch 前执行:

1. [参数验证]  参数格式/范围/类型检查
2. [权限检查]  当前操作在哪个权限等级
3. [路径检查]  文件路径是否在安全白名单内
4. [命令检查]  系统命令是否在黑名单中
5. [速率限制]  防止短时间内大量操作
              (如 1 秒内不超过 3 次点击)
```

### 数据隐私保护

```
├── 摄像头: 帧捕获后立即处理，不保存到磁盘
├── 录音: 识别后丢弃原始音频，只保留文本
├── 截图: 临时内存中，不自动保存
├── 记忆: 明确的记忆存储/删除指令
└── 日志: 不记录摄像头/麦克风原始数据
```

## 六、模块间通信协议

### 内部消息格式

所有模块间通信通过统一的消息总线：

```python
# 内部消息 (模块间)
@dataclass
class InternalMessage:
    type: str                    # "input.voice" / "input.camera" / "tool.result"
    source: str                  # 来源模块名
    target: str                  # 目标模块名
    payload: dict                # 消息体
    timestamp: float             # 时间戳
    
# Agent 内部决策
@dataclass
class AgentDecision:
    action: str                  # "speak" / "tool_call" / "wait"
    content: str | None          # 要说的内容
    tool_name: str | None        # 要调用的工具
    tool_params: dict | None     # 工具参数
```

## 七、插件/工具扩展机制

### 添加新工具的方式

```python
# tools/my_tool.py
from core.tool_registry import register_tool

@register_tool(
    name="weather_query",
    description="查询指定城市的天气",
    category="web",
    parameters={
        "type": "object",
        "properties": {
            "city": {
                "type": "string",
                "description": "城市名称, 如 '北京'"
            }
        },
        "required": ["city"]
    }
)
async def weather_query(city: str) -> dict:
    """查询天气的实现"""
    # ... 调用天气 API
    return {"success": True, "data": {"city": city, "temp": 25}}
```

只需把文件放到 `tools/` 目录下，工具注册器自动发现。

### 工具自动化注册

```
tools/ 目录扫描:
  └── *.py 文件 → 检查 @register_tool 装饰器
      → 自动注册到 tool_registry
      → 工具描述自动注入 LLM System Prompt
      → LLM 在 ReAct 中自动知晓可用工具
```

## 九、补充设计决策

### 为什么用 asyncio 而非多线程？
- Python GIL 让多线程无法真正并行 CPU 任务
- asyncio 的协作式调度更适合 IO 密集型 AI 应用
- 阻塞操作 (Whisper/OCR) 用线程池隔离

### 为什么摄像头和麦克风保持独立？
- 摄像头只在用户明确要求时才捕获
- 麦克风持续监听但只保存环形缓冲
- 唤醒前不进行任何 AI 处理，保护隐私

### TTS 播放时还能听到用户说话吗？
- 利用 Edge-TTS 流式播放 + 麦克风同时监听
- 检测到用户说话阈值 → 立即停止 TTS → 进入 LISTEN
- 类似人类对话的自然打断体验
